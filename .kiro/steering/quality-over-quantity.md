# 品質優先原則（從 250 天失敗中學到的）

**這份準則來自 DAY-290 的誠實反思。優先級高於所有其他自主運作原則。**

---

## 原則一：玩一局才算開始

每次自主循環的**第一步**是玩一局（或看玩家錄影），不是加功能。

```
❌ 錯誤：想到功能 → 實作 → build 通過 → 宣稱完成
✅ 正確：玩一局 → 找最差的地方 → 修那個地方 → 再玩一局確認
```

沒有玩過就沒有資格說「完成了」。

---

## 原則二：禁止宣稱「完成度 100%」

「完成度 100%」是一個陷阱指標，宣稱後所有工作都失去方向感。

**改用這些指標：**
- 射擊手感：玩家點擊後的反饋是否即時且爽快？（0-10）
- 視覺清晰度：玩家能在 1 秒內識別高價值目標嗎？（0-10）
- 核心循環流暢度：射擊→擊破→獎勵→再射擊 是否無縫？（0-10）
- 特效密度：每 30 秒有幾次讓玩家「哇」的時刻？

---

## 原則三：三層驗證缺一不可

每個新功能必須通過三層驗證才算完成：

| 層次 | 驗證方式 | 不能跳過的原因 |
|------|---------|--------------|
| Server 編譯 | `go build ./...` + `go vet ./...` | 基本正確性 |
| Client 顯示 | 實際在 Godot 中看到效果 | 編譯通過 ≠ 顯示正確 |
| 玩家感受 | 錄影或親自遊玩 | 顯示正確 ≠ 體驗好 |

**`go build` 通過只是第一層，不是終點。**

---

## 原則四：深度優先於廣度

一個好的捕魚機有 20-40 個精心設計的目標物，不是 248 個。

```
❌ 錯誤：加第 249 個幸運魚系統
✅ 正確：讓現有的 10 個最常出現的目標物，每個都有完整的視覺特效、音效、手感
```

**問自己：「如果只能改一件事，最影響玩家體驗的是什麼？」**
然後去改那件事，不是加新功能。

---

## 原則五：知識庫是健康指標

如果 knowhow-log.md 超過 **2 週沒有更新**，代表工作模式出了問題：
- 可能在重複做同樣的事
- 可能沒有遇到真實問題（因為沒有真實測試）
- 可能在自我強化而不是真正進步

每次遇到任何問題（不管多小），都要記錄到 knowhow-log。

---

## 原則六：影片分析的正確方法

當拿到玩家錄影時，要找：
1. **前 30 秒**：玩家第一反應是什麼？
2. **停頓點**：玩家在哪裡猶豫或困惑？
3. **爽感點**：玩家在哪裡有明顯正面反應？
4. **時機問題**：特效和音效是否在正確時機出現？

不是確認「功能有沒有出現」，而是「感受玩家的情緒流動」。

---

## 原則八：體驗循環（v2.0 架構核心）

每個功能必須走完以下循環才算完成：

```
設計層確認 → 實作層開發 → 整合層驗證 → 驗證層評估
```

對應 Agent：
- 整合層：Integration Test Agent + Protocol Sync Agent（不可跳過）
- 驗證層：QA Agent + Player Experience Agent + Regression Guard Agent（不可跳過）

**跳過整合層或驗證層 = 功能不算完成。**

---

## 原則七：UI 腳本數量警戒線

- 超過 **20 個同類型腳本**（如 LuckyXxxPanel.gd）→ 必須建立基礎類別
- 超過 **50 個 UI 腳本** → 必須做架構重構，不能繼續加新腳本
- 目前狀態：150+ 個 UI 腳本，**已超過警戒線**，下一個重要任務是重構

---

## 強制自問（每次完成任何工作後）

> **「我最後一次玩這個遊戲是什麼時候？玩的時候最讓我不爽的是什麼？我修了嗎？」**

如果答案是「我沒有玩過」或「我沒有修」，那就不算完成。
