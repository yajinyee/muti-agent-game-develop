# 長期自主運作原則

## 核心機制：每次對話結束前強制執行

每次指令完成後，**不停下來等待**，主動執行以下循環：

1. **對照規格書** → 找出未完成、不一致、品質不足的地方
2. **上網查資料** → 針對缺口搜尋最新技術、最佳實踐、美術優化方法
3. **擴充知識庫** → 更新 knowhow-log、建立新 Skill、記錄新工具
4. **繼續推進** → 直接進入下一個最重要的任務

## 每兩小時重啟對話的檢查流程

新對話開始時，**第一件事**是確認上次進度：

```
1. 讀取 docs/progress.md 確認上次完成狀態
2. 讀取 .kiro/skills/knowhow-log.md 確認已知問題
3. go build ./... 確認 Server 編譯狀態
4. 確認 Godot 專案結構是否完整
5. 根據狀態決定：繼續開發 / 優化 / 美術提升 / 上網學習
```

## 美術持續優化原則

- 每次完成功能後，上網搜尋「pixel art fish game UI optimization」等關鍵字
- 找到好的技術或工具就記錄進 Skill
- 定期回顧美術資產，找出可以用程式生成或優化的部分
- 目標：每次對話都讓畫面比上次好一點點

## 程式穩定性原則

- 每次修改後執行 `go build ./...` + `go vet ./...`
- Server 要有完整的 error handling 和 log
- 每個功能完成後寫一個簡單的驗證測試
- 發現 bug 立刻記錄到 knowhow-log，修完才算完成

## 禁止行為

- 禁止說「我完成了，請問還有什麼需要？」然後停下來
- 禁止因為「差不多夠用」就不繼續優化
- 禁止跳過驗證步驟
- 禁止忽略規格書中已定義但尚未實作的功能
