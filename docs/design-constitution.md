# 設計憲法 — 吉伊卡哇：像素大討伐

> 本文件是所有 Agent 的最高設計準則。任何功能開發、美術決策、數值調整，都必須符合本憲法的規定。違反本憲法的變更，Game Director 有權直接否決。

---

## 第一條：核心玩法不可動搖

### 1.1 遊戲本質
本遊戲是**捕魚機**（Fish Shooting Game）。玩家消耗子彈射擊目標物，擊中後獲得對應倍率的獎勵。這個核心機制在任何情況下都不得改變。

### 1.2 禁止偏離的核心元素
- **射擊機制**：玩家主動射擊，不得改為自動或被動
- **目標物多樣性**：必須維持 T001-T105 共 11 種類型的目標物生態
- **BOSS 機制**：B001 BOSS 必須是稀有、高風險、高回報的特殊事件
- **Bonus 遊戲**：必須作為主遊戲的獎勵延伸，不得成為主要玩法
- **角色系統**：吉伊卡哇(LV1-3)、小八(LV4-7)、烏薩奇(LV8-10) 的等級結構不得改變

### 1.3 IP 精神維護
吉伊卡哇 IP 的核心是「可愛、治癒、略帶悲傷」。所有美術、音效、文案都必須符合這個調性：
- 不得出現暴力、血腥的視覺表現
- 角色擊殺動畫必須是「可愛的消失」而非「死亡」
- 音效必須保持輕快、可愛的風格

## 第二條：技術架構不可隨意更換

### 2.1 固定技術棧
| 層級 | 技術 | 版本 | 理由 |
|------|------|------|------|
| 遊戲引擎 | Godot | 4.6.2 | 已完成大量開發，不得降版 |
| 伺服器語言 | Go | 1.21+ | 效能與穩定性 |
| 通訊協定 | WebSocket | RFC 6455 | 已完成實作 |
| 匯出平台 | HTML5 | Web 標準 | 目標部署環境 |
| 伺服器 Port | 7777 | - | 遊戲業界標準 |

### 2.2 禁止的技術決策
- 禁止將 Go Server 改為其他語言（Node.js、Python 等）
- 禁止將 WebSocket 改為 HTTP Polling 或其他協定
- 禁止降級 Godot 版本
- 禁止更換 Port（除非有充分理由並經 Game Director 批准）

### 2.3 允許的技術擴展
- 可以新增輔助工具（Python 腳本、Go CLI 工具）
- 可以新增 Go 套件（需記錄到 go.mod）
- 可以新增 Godot 插件（需測試 HTML5 相容性）

## 第三條：數值設計邊界

### 3.1 RTP 安全範圍
- **最低 RTP**：88%（低於此值玩家體驗極差）
- **最高 RTP**：98%（高於此值商業不可行）
- **目標 RTP**：92-96%（最佳平衡點）
- **Bonus 遊戲 RTP**：105-115%（補償機制，允許超過 100%）

### 3.2 遊戲節奏邊界
- **Bonus 觸發頻率**：每 50-100 局（不得低於 50 或高於 150）
- **BOSS 出現頻率**：每 200-500 局（不得低於 100 或高於 1000）
- **最大單次獎勵**：500x 下注金額（防止極端波動）
- **最小有效射擊距離**：不限（玩家可自由選擇目標）

### 3.3 禁止的數值操作
- 禁止讓 RTP 在短期內劇烈波動（單次調整不得超過 ±3%）
- 禁止讓 BOSS 出現頻率低於每 100 局（玩家等待感過強）
- 禁止讓任何單一目標物的 RTP 貢獻超過 50%（過度依賴）

## 第四條：美術品質標準

### 4.1 像素風格規範
- **風格**：16-bit 像素藝術，參考吉伊卡哇官方像素周邊風格
- **調色板**：每個角色最多 32 色，目標物最多 16 色
- **輪廓**：必須有 1px 黑色輪廓（增加辨識度）
- **解析度**：角色 64-128px，目標物 64px，BOSS 256px

### 4.2 視覺一致性要求
- 所有角色必須使用相同的光源方向（左上方）
- 所有目標物必須有相同的「被擊中」視覺反饋（閃白）
- 背景與前景的對比度必須足夠（目標物清晰可見）

### 4.3 禁止的美術決策
- 禁止使用寫實風格（必須保持像素藝術）
- 禁止讓角色設計偏離吉伊卡哇 IP 的外觀特徵
- 禁止使用超過 32 色的調色板（破壞像素風格）
- 禁止在未通過 Art Director 審核的情況下替換正式素材

---

## 第五條：玩家體驗優先原則

### 5.1 回饋即時性
- 射擊後的視覺/音效回饋必須在 100ms 內出現
- 獎勵顯示必須清晰、醒目
- 所有 UI 操作回應時間 < 50ms

### 5.2 清晰度原則
- 玩家必須能清楚看到目標物的移動路徑
- 獎勵倍率必須在目標物上或附近清晰顯示
- 當前下注金額必須隨時可見

### 5.3 公平感原則
- 射擊判定必須基於伺服器端計算（防作弊）
- 不得有任何視覺欺騙（顯示命中但實際未命中）
- RTP 必須在長期統計下符合設定值

---

## 第六條：變更管理

### 6.1 需要 Game Director 批准的變更
- 任何核心玩法機制的修改
- RTP 調整超過 ±1%
- 新增或移除角色/目標物類型
- 技術架構的重大變更
- 品質門檻的調整

### 6.2 需要 Spec Architect 審核的變更
- WebSocket 協定的任何修改
- 新增 API 端點或訊息類型
- 資料結構的變更

### 6.3 可以自主執行的變更
- Bug 修復（不影響核心玩法）
- 美術資產的品質提升（在現有規格內）
- 效能優化（不改變功能行為）
- 文件更新
