# 品質驗收標準 — 吉伊卡哇：像素大討伐

> 本文件定義所有品質門檻與驗收條件。所有 Agent 必須在完成工作後對照本文件進行自我評估。QA Agent 負責最終驗收。

---

## 品質分數總覽

| 指標 | 當前分數 | 門檻 | 狀態 | 硬規則 |
|------|---------|------|------|-------|
| Spec Completeness | 95 | >= 95 | ✅ | 低於門檻停止新功能 |
| Build Stability | - | >= 95 | 待測 | 低於門檻禁止展示版 |
| Visual Consistency | 91 | >= 90 | ✅ | 低於門檻禁止替換素材 |
| Animation Quality | - | >= 88 | 待測 | 低於門檻禁止 merge |
| Audio Sync | - | >= 90 | 待測 | 低於門檻重新調整 |
| Gameplay Feel | - | >= 85 | 待測 | 低於門檻優先修復 |
| Balance Health | - | >= 90 | 待測 | 低於門檻重新模擬 |
| Regression Risk | - | <= 10 | 待測 | 超過門檻自動 rollback |

---

## 1. Spec Completeness（規格完整性）

### 定義
所有已定義的功能規格都有對應的實作，且實作與規格一致的比例。

### 評分標準
| 分數 | 條件 |
|------|------|
| 100 | 所有規格都有實作，且完全一致 |
| 95-99 | 少數非核心規格未實作或有小差異 |
| 90-94 | 部分規格未實作，但核心功能完整 |
| < 90 | 核心功能有規格缺失或不一致 |

### 驗收項目
- [ ] WebSocket 訊息類型全部實作（shoot/hit/miss/bonus/boss）
- [ ] 目標物 T001-T105 全部定義且可生成
- [ ] B001 BOSS 完整實作（生成/攻擊/死亡）
- [ ] 角色三種（吉伊卡哇/小八/烏薩奇）全部可用
- [ ] Bonus 遊戲完整流程
- [ ] 下注系統（最小/最大/調整）
- [ ] 分數顯示與結算

### 硬規則
- **Spec Completeness < 95**：停止所有新功能開發，優先補齊規格

---

## 2. Build Stability（建置穩定性）

### 定義
在標準測試流程中，遊戲能夠正常啟動、運行、結束，且不發生崩潰或嚴重錯誤的比例。

### 評分標準
| 分數 | 條件 |
|------|------|
| 100 | 所有測試案例通過，零崩潰 |
| 95-99 | 極少數邊界案例失敗，核心流程穩定 |
| 90-94 | 偶發性問題，但可重現率低 |
| < 90 | 頻繁崩潰或核心功能不穩定 |

### 驗收項目
- [ ] Go Server 啟動無錯誤（`go build ./...` 通過）
- [ ] Godot HTML5 匯出成功
- [ ] WebSocket 連線建立成功
- [ ] 連續遊玩 30 分鐘無崩潰
- [ ] 斷線重連機制正常
- [ ] 100 個目標物同時在場不崩潰
- [ ] BOSS 戰完整流程不崩潰

### 硬規則
- **Build Stability < 95**：禁止產出展示版（Demo Build）

---

## 3. Visual Consistency（視覺一致性）

### 定義
所有美術資產在風格、色彩、像素密度、輪廓風格上的一致程度。

### 評分標準
| 分數 | 條件 |
|------|------|
| 100 | 所有資產完全統一，無任何風格衝突 |
| 90-99 | 極少數資產有輕微差異，整體感覺統一 |
| 80-89 | 部分資產風格不一致，但不影響整體觀感 |
| < 80 | 明顯的風格衝突，破壞遊戲視覺體驗 |

### 驗收項目
- [ ] 所有角色使用相同的像素密度（64-128px）
- [ ] 所有目標物有 1px 黑色輪廓
- [ ] 調色板限制：角色 <= 32 色，目標物 <= 16 色
- [ ] 光源方向一致（左上方）
- [ ] 背景與目標物對比度足夠（目標物清晰可見）
- [ ] BOSS 視覺差異化明顯（尺寸、色彩、特效）

### 硬規則
- **Visual Consistency < 90**：禁止替換正式素材，必須重新生成

---

## 4. Animation Quality（動畫品質）

### 定義
所有動畫的流暢度、同步精度、視覺表現的綜合評分。

### 評分標準
| 分數 | 條件 |
|------|------|
| 100 | 所有動畫流暢、同步精確、視覺表現優秀 |
| 88-99 | 動畫整體流暢，少數細節可優化 |
| 80-87 | 部分動畫有卡頓或同步問題 |
| < 80 | 明顯的動畫問題，影響遊戲體驗 |

### 驗收項目
- [ ] 角色 idle 動畫無縫循環（首尾幀一致）
- [ ] 攻擊動畫與音效同步（誤差 < 50ms）
- [ ] 目標物游泳動畫流暢（最少 4 幀，FPS 8-12）
- [ ] 角色攻擊動畫完整（最少 6 幀，FPS 12-24）
- [ ] BOSS 進場動畫有衝擊感
- [ ] 目標物消滅動畫有滿足感
- [ ] 60 FPS 下所有動畫流暢播放

### 硬規則
- **Animation Quality < 88**：禁止 merge 到主分支

---

## 5. Audio Sync（音效同步）

### 定義
音效觸發時機與遊戲事件的同步精度評分。

### 評分標準
| 分數 | 條件 |
|------|------|
| 100 | 所有音效與事件完美同步（< 16ms 誤差）|
| 90-99 | 絕大多數音效同步良好（< 50ms 誤差）|
| 80-89 | 部分音效有明顯延遲（50-100ms）|
| < 80 | 音效嚴重不同步，破壞遊戲體驗 |

### 驗收項目
- [ ] 攻擊音效與攻擊動畫同步（< 50ms）
- [ ] 命中音效與碰撞事件同步（< 50ms）
- [ ] BOSS 警告音效在 BOSS 出現前 2 秒觸發
- [ ] Bonus 音效在 Bonus 觸發時立即播放
- [ ] 背景音樂無縫循環（無明顯接縫）
- [ ] 音量標準化（BGM -14 LUFS，音效 -6 LUFS）

### 硬規則
- **Audio Sync < 90**：必須重新調整觸發時機

---

## 6. Gameplay Feel（遊戲手感）

### 定義
玩家操作的即時回饋感、遊戲節奏感、整體遊玩體驗的主觀評分。

### 評分標準
| 分數 | 條件 |
|------|------|
| 100 | 操作爽快、節奏感強、讓人想繼續玩 |
| 85-99 | 整體手感良好，少數細節可優化 |
| 70-84 | 基本可玩，但缺乏爽快感 |
| < 70 | 操作遲鈍或節奏感差，影響遊玩意願 |

### 驗收項目
- [ ] 射擊後視覺回饋 < 100ms
- [ ] 命中特效有衝擊感（閃白、粒子效果）
- [ ] 獎勵顯示清晰醒目（大字體、動畫）
- [ ] BOSS 戰有緊張感（音效、視覺效果）
- [ ] Bonus 遊戲有興奮感（特殊音效、視覺）
- [ ] 遊戲節奏不會讓玩家感到無聊（目標物密度適中）

### 硬規則
- **Gameplay Feel < 85**：優先修復玩法問題，暫停其他優化

---

## 7. Balance Health（平衡健康度）

### 定義
遊戲數值設計的合理性，包含 RTP 準確性、獎勵分佈、遊戲節奏的綜合評分。

### 評分標準
| 分數 | 條件 |
|------|------|
| 100 | RTP 精確、獎勵分佈合理、節奏完美 |
| 90-99 | 整體平衡良好，少數數值可微調 |
| 80-89 | 部分數值偏差，但在可接受範圍 |
| < 80 | 明顯的數值問題，影響商業可行性或玩家體驗 |

### 驗收項目
- [ ] RTP 模擬結果在 92-96% 範圍內（100 萬局）
- [ ] RTP 標準差 < ±2%
- [ ] Bonus 觸發頻率：每 50-100 局
- [ ] BOSS 出現頻率：每 200-500 局
- [ ] 無任何單一目標物 RTP 貢獻超過 50%
- [ ] 最大單次獎勵 <= 500x

### 硬規則
- **Balance Health < 90**：必須重新模擬並調整數值

---

## 8. Regression Risk（回歸風險）

### 定義
本次變更導致既有功能出現問題的風險評分（越低越好）。

### 評分標準
| 分數 | 條件 |
|------|------|
| 0-5 | 極低風險，變更範圍小且有完整測試 |
| 6-10 | 低風險，有少量潛在影響但已測試 |
| 11-20 | 中風險，需要額外關注 |
| > 20 | 高風險，建議 rollback |

### 評估因素
- 修改的程式碼行數
- 修改的模組數量
- 是否涉及核心邏輯（RTP、WebSocket、遊戲狀態機）
- 測試覆蓋率
- 是否有對應的回歸測試

### 硬規則
- **Regression Risk > 10**：自動觸發 rollback 警告，需 Game Director 確認才能繼續

## 整體驗收流程

```
1. 開發完成
   ↓
2. 各 Agent 自我評估（對照本文件）
   ↓
3. QA Agent 執行完整測試
   ↓
4. 計算所有品質分數
   ↓
5. 所有分數達標？
   ├── 是 → 標記為 Build Stable，可產出展示版
   └── 否 → 找出最低分項目，指派對應 Agent 修復
              ↓
              回到步驟 1
```
